#软件架构设计

## 硬件和平台解耦

实现需求：

1. 更换硬件连接，应用层无需改变，只需修改board_config(板级抽象)

2. 更换芯片，业务代码无需改变，只需做interface对应平台实现和移植


一个成熟、解耦的软件架构，核心在于**“依赖倒置”**和**“接口隔离”**。高层模块(稳定)不应该依赖于低层模块(变化)，二者都应该依赖于抽象(稳定)。抽象(稳定)不应该依赖于实现细节(变化），实现细节应该依赖于抽象(稳定)。使用封装来创建对象之间的分界层，让设计者可以在分界层的一侧进行修改，而不会对另一侧产生不良的影响，从而实现层次间的松耦合。

我们要实现的目标架构如下：

| 层级                      | 对应目录                 | 主要职责                                                     | 依赖关系                            |
| :------------------------ | :----------------------- | :----------------------------------------------------------- | :---------------------------------- |
| **应用层 (APP/Core)**     | `Core/`, `App/`, Test/   | 实现最终业务逻辑（如 `main` 循环、具体测试用例 test_led.c、UI 控制）。 | 依赖 **接口层** 和 **组件层**。     |
| **组件层 (Component)**    | `Component/`             | 提供通用的软件功能（如`uart_queue` 、`ring_buffer` 等）。    | 依赖 **接口层**。                   |
| **接口层 (Interface)**    | `BSP/Drivers/interface/` | **核心解耦层**。定义硬件无关的抽象接口（如 `Sys.h`, `MemPool.h`, `usart_driver.h`）。 | **无硬件依赖**。                    |
| **BSP 管理/工厂层**       | `BSP/Drivers/factory/`   | 负责硬件实例的注册与分发。将具体的芯片驱动绑定到抽象接口句柄（如 `USART_ID_DEBUG`）。 | 依赖 **接口层** 和 **平台实现层**。 |
| **平台实现层 (Platform)** | `BSP/Drivers/platform/`  | 针对特定芯片（如 `stm32`）实现接口层的 API，直接调用芯片厂商的 HAL/库。 | 依赖 **接口层** 和 **厂商 HAL**。   |
| **硬件抽象层 (HAL/DRV)**  | `Core/Src/`, `Drivers/`  | 厂商提供的原始驱动（STM32Cube HAL）。                        | 依赖具体硬件寄存器。                |

### 例：创建一个串口队列，并做stm32、ch32的实现

#### 1. 组件(Component)

串口队列的FIFO一般使用**环形缓冲**，环形队列（也称循环队列）是一种特殊的队列，它是一个固定大小的数组，通过特定的算法使得队列的头尾在内存上实现连接。环形队列的特点是当队列满时，新的数据会覆盖最早的数据，从而实现循环利用内存无需手动释放。但当负载过大(写指针超过读指针)时，如一次发送很多日志信息，会造成数据覆盖，导致较早的信息丢失损坏，此时可增大缓冲区大小或做其他适当处理。
此模块是纯算法组件不依赖硬件，在Component文件夹中创建.h和.c实现，基本致结构如下：

```c
// 环形缓冲区
typedef struct {
  uint8_t *buffer;
  size_t size;
  volatile size_t head; // 写指针
  volatile size_t tail; // 读指针
} ring_buffer_t;
ring_buffer_t* rb_create(uint8_t *buffer, size_t size);
```

#### 2. 驱动接口(interface)

**接口**是一种抽象类型，用于定义一组方法规范，而不提供具体实现。它的核心作用是为类的行为制定契约，确保实现接口的类遵循统一的规范。在**interface**文件夹中创建**usart_driver.h**，基本结构如下：

```c
// 事件类型定义
typedef enum{
  USART_EVENT_RX_COMPLETE,
  USART_EVENT_TX_COMPLETE,
  USART_EVENT_ERROR,
  USART_EVENT_RX_EVENT,
} usart_event_t;

// 回调函数原型
// context:上下文, event:事件类型, args:事件参数(如接收到的数据长度等)
typedef void (*usart_callback_t)(void *context, usart_event_t event, void *args);

// USART 操作接口
typedef struct{
  int (*transmit)(usart_driver_t *self, const uint8_t *data, size_t size,uint32_t timeout);
  int (*receive)(usart_driver_t *self, uint8_t *buffer, size_t size,uint32_t timeout);
  int (*transmit_asyn)(usart_driver_t *self, const uint8_t *data, size_t size);
  int (*receive_asyn)(usart_driver_t *self, uint8_t *buffer, size_t size);
  int (*set_callback)(usart_driver_t *self, usart_callback_t cb, void *cb_context);
} usart_driver_ops_t;

// USART 驱动基类
struct usart_driver_t{
  const usart_driver_ops_t *ops;
};
```

以下详细说明**stm32_usart_driver.c**的实现：

###### **基本操作实现：**

```c
// 同步发送函数
static int stm32_usart_transmit(usart_driver_t *self, const uint8_t *data,
                                size_t size, uint32_t timeout) {
  stm32_usart_driver_t *driver = (stm32_usart_driver_t *)self; //将从接口接收的通用驱动转回平台驱动
  // 使用阻塞模式发送
  if (HAL_UART_Transmit(driver->huart,(uint8_t *)data,(uint16_t)size,timeout) == HAL_OK) {return 0;}
  return -1;
}

// 同步接收函数
static int stm32_usart_receive(usart_driver_t *self,uint8_t *buffer,size_t size,uint32_t timeout) {
  stm32_usart_driver_t *driver = (stm32_usart_driver_t *)self;
  HAL_StatusTypeDef status = HAL_UART_Receive(driver->huart, buffer, (uint16_t)size, timeout);
  if (status == HAL_OK) { return 0; }
  return -1;
}

// 异步发送函数（使用DMA实现）
static int stm32_usart_transmit_asyn(usart_driver_t *self, const uint8_t *data, size_t size) {
  stm32_usart_driver_t *driver = (stm32_usart_driver_t *)self;
  if (driver->T_isbusy) { return -1; }
  driver->T_isbusy = 1;
  if (driver->callback == NULL) {
    driver->T_isbusy = 0;
    return -1;
  }
  // 使用DMA模式发送
  if (HAL_UART_Transmit_DMA(driver->huart, (uint8_t *)data, (uint16_t)size) == HAL_OK) { return 0; }
  driver->T_isbusy = 0;
  return -1;
}

// 异步接收函数（使用DMA）
static int stm32_usart_receive_asyn(usart_driver_t *self, uint8_t *buffer, size_t size) {
  stm32_usart_driver_t *driver = (stm32_usart_driver_t *)self;
  if (driver->R_isbusy) { return -1; }
  driver->R_isbusy = 1;
  if (driver->callback == NULL) {
    driver->R_isbusy = 0;
    return -1;
  }
  // 使用DMA模式接收
  if (HAL_UARTEx_ReceiveToIdle_DMA(driver->huart, buffer, (uint16_t)size) == HAL_OK) { return 0; }
  driver->R_isbusy = 0;
  return -1;
}

// stm32 USART 驱动函数表
static const usart_driver_ops_t stm32_usart_ops = {
    .transmit = stm32_usart_transmit,
    .receive = stm32_usart_receive,
    .transmit_asyn = stm32_usart_transmit_asyn,
    .receive_asyn = stm32_usart_receive_asyn,
    .set_callback = stm32_usart_set_callback,
};
```

###### **中断注册实现：**

接口定义了可外部注册的回调函数，最终我们需要在**平台的回调函数**中找到**触发本次中断的对象**的回调函数来调用，但HLA库的**中断上下文**只传递像**UART_HandleTypeDef *huart**这样的硬件句柄，为此我们需要保存所有创建的**驱动对象的地址**到一个表中，在平台回调中找回此句柄对应的驱动对象。实现如下：

```c
#define MAX_USART_INSTANCES 4 // 最大可创建的对象数量
static stm32_usart_driver_t *g_usart_instances[MAX_USART_INSTANCES] = {0};	// 驱动表
// 实例注册
static void register_instance(stm32_usart_driver_t *instance) {
  for (int i = 0; i < MAX_USART_INSTANCES; i++) {
    if (g_usart_instances[i] == NULL) {
      g_usart_instances[i] = instance;
      return;
} } }
// 回调注册
static int stm32_usart_set_callback(usart_driver_t *base, usart_callback_t cb, void*ctx) {
  stm32_usart_driver_t *self = (stm32_usart_driver_t *)base;
  self->callback = cb;
  self->cb_context = ctx; //用户上下文
  return 0;
}
// 驱动创建
stm32_usart_driver_t *stm32_usart_driver_create(UART_HandleTypeDef *huart) {
  stm32_usart_driver_t *driver =
    (stm32_usart_driver_t *)sys_malloc(USART_MEMSOURCE, sizeof(stm32_usart_driver_t));//从内存池分配
  if (driver) {
    driver->base.ops = &stm32_usart_ops;
    driver->huart = huart;
    driver->callback = NULL;
    driver->cb_context = NULL;
    driver->T_isbusy = 0;
    driver->R_isbusy = 0;
    register_instance(driver);} // 在创建时将此驱动添加到表中
  return driver;
}
// 中断分发处理
static void dispatch_irq(UART_HandleTypeDef *huart, usart_event_t event, void *args) {
  for (int i = 0; i < MAX_USART_INSTANCES; i++) {
    if (g_usart_instances[i] && g_usart_instances[i]->huart == huart) { // 寻找此次中断的所属对象
      stm32_usart_driver_t *driver = g_usart_instances[i]; // 获取stm32_usart_driver_t对象
      // 找回驱动对象driver后就可以访问它的所有资源了
      if (event == USART_EVENT_RX_COMPLETE || event == USART_EVENT_RX_EVENT) { driver->R_isbusy = 0;}
      if (event == USART_EVENT_TX_COMPLETE) { driver->T_isbusy = 0; }
      if (driver->callback) {
        driver->callback(driver->cb_context, event, args); // 调用对象的回调函数(如果有)
      }
      break;
} } }
// HAL 库回调函数重写
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart) {
  dispatch_irq(huart, USART_EVENT_RX_COMPLETE, NULL); // 接收完成事件
}
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart) {
  dispatch_irq(huart, USART_EVENT_TX_COMPLETE, NULL); // 发送完成事件
}
void HAL_UARTEx_RxEventCallback(UART_HandleTypeDef *huart, uint16_t Size) {
  dispatch_irq(huart, USART_EVENT_RX_EVENT, (void *)Size); // 接收事件
}
void HAL_UART_ErrorCallback(UART_HandleTypeDef *huart) {
  dispatch_irq(huart, USART_EVENT_ERROR, NULL);	// 错误
}
```

#### 3. 工厂/Board层

不同平台的驱动创建方式可能不同，工厂模式提供了一种创建对象的方式，使得创建对象的过程与使用对象的过程分离。此层连接**板级配置**与**平台实现**的关键枢纽。它通过统一的接口 `usart_driver_get()` 为上层应用提供硬件实例，同时隐藏了底层硬件和平台的差异性。

其核心工作流程如下：

1.  **请求处理**：上层应用通过逻辑ID（如 `USART_ID_DEBUG`）调用 `usart_driver_get(id)` 请求一个串口驱动实例。
2.  **设备查找**：工厂层检查内部缓存数组 `usart_drivers[id]`。如果该ID的驱动实例已创建，则直接返回；否则进入创建流程。
3.  **资源映射**：工厂层通过 `id` 在全局的 `usart_mappings` 映射表中查找对应的物理资源 `resource`。这个映射表在 `BSP/Board/` 目录下定义，将逻辑ID与具体的硬件句柄（如 `&huart1`）关联起来。
4.  **平台适配**：根据项目配置（`factory_config.h` 中的 `USART_DRIVER_PLATFORM`），工厂层调用对应平台的创建函数。例如，在STM32平台上，会调用 `stm32_usart_driver_create((UART_HandleTypeDef *)mapping->resource)`，并将从映射表中获取的 `resource` 强转为 `UART_HandleTypeDef*` 类型传入。
5.  **实例化与缓存**：平台特定的创建函数负责分配内存、初始化平台相关的数据结构，并最终返回一个符合抽象接口 `usart_driver_t` 的指针。工厂层将此指针缓存在 `usart_drivers` 数组中，以供后续请求复用，并将其返回给调用者。

这种结构实现了很好的解耦：
*   **应用层** 只需关心“我要调试串口”，而无需知道它实际连接到MCU的哪个引脚或使用哪个外设实例（USART1还是USART2）。
*   **板级配置** 只需修改 `dev_map.h` 和 `dev_map.c` 中的映射关系，即可改变硬件布局，而无需改动任何应用代码。
*   **平台移植** 只需实现新的 `xxx_usart_driver_create` 函数并添加到工厂函数中，即可将整个系统移植到新平台上。

##### **代码示例**

以下是关键的代码片段：

**a) 设备映射表 (dev_map.h & stm32h743_board_v1.c)**

```c
// dev_map.h: 定义逻辑ID
typedef enum {
  USART_ID_DEBUG = 0,   // 日志和调试
  USART_ID_ESP8266,     // 网络通信模块
  USART_MAX_DEVICES
} usart_device_id_t;
// 定义映射项结构（屏蔽平台差异）
typedef struct {
  void *resource; // 物理资源
} usart_mapping_t;
// 声明映射表
extern const usart_mapping_t usart_mappings[USART_MAX_DEVICES];

// stm32h743_board_v1.c:
// 定义物理映射表，通过不同的.c文件还可创建不同版本的硬件映射
const usart_mapping_t usart_mappings[USART_MAX_DEVICES] = {
    [USART_ID_DEBUG] = {.resource = (void *)&huart1},
    [USART_ID_ESP8266] = {.resource = (void *)&huart2},
};
```

**b) 工厂实现 (usart_factory.c)**

```c
// 通过此函数获取usart硬件驱动
usart_driver_t *usart_driver_get(usart_device_id_t id) {
  if (id >= USART_MAX_DEVICES)
    return NULL;
  if (usart_drivers[id] == NULL) {
    const usart_mapping_t *mapping = &usart_mappings[id];
    if (mapping->resource == NULL)
      return NULL;
// 通过宏切换平台
#if (USART_DRIVER_PLATFORM == PLATFORM_STM32) // stm32
    usart_drivers[id] = (usart_driver_t *)stm32_usart_driver_create(
        (UART_HandleTypeDef *)mapping->resource);  // 调用平台的接口创建
#elif (USART_DRIVER_PLATFORM == PLATFORM_CH32V003) // ch32
    usart_drivers[id] = (usart_driver_t *)ch32v003_usart_driver_create(
        (USART_TypeDef *)mapping->resource);
#endif
  }
  return usart_drivers[id];
}
```
