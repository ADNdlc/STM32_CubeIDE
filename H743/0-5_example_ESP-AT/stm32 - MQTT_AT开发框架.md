# stm32 - MQTT_AT开发框架

​	**此工程重构自 IDE/F103/F103C8T6_ESP32AT 实现的基本功能没有什么不同。是为了解决原方案 “发送 - 等待” 模式和 异步通信 云事件 的冲突，让发送与接收解耦**。

> [AT 消息类型和命令分类](https://docs.espressif.com/projects/esp-at/zh_CN/latest/esp32/AT_Command_Set/index.html#id11)

## 串口通信层

​	基本没有改动，为了不遗漏模块回复的每条消息。每次接受完需要尽可能快地重启接收，修改了**环形缓冲区**管理方案，**原方案**为**DMA普通模式**接收，在**中断中**将数据写入环形缓冲区，自己管理缓冲区的读写回绕，解析时从**loopbuf**读出到**cmd_buffer**进行字符串匹配。
​	**现方案**删去第一层拷贝，使用**DMA环形模式**直接向**loopbuf**写数据，所以只需要管理读指针即可，现在uart通信层只负责接收模块发来的字节流，并对外提供读函数，不再关心通信内容。简化了通信层整体结构和职责并提高了通信效率和稳定性。
​	对外提供 **缓冲区读取函数**，**获取未读字节数函数**。

## 行解析层

​	**新增功能**，负责拆包。模块对部分命令回复可能包含不止一条消息，需要将每行消息通过结尾的“ \r\n”拆分开单独解析。**原方案**循环等待并匹配字符串，本质上将命令发送和响应解析作为一个整体所以没有此需求。**现方案**将命令发送，消息解析，拆分为两个互不干扰的任务需要明确每一条回复消息的含义来做**响应分发**。
​	此层接收外部输入数据，从中**拆分出一个完整行**(以“ \r\n”为结尾) 或 字符“>”(此回复没有换行)后在回调中传给**响应分发器**。同时保证非完整行的保留与正确拼接。
​	对外只提供 **AT_parser_input** 数据输入接口。

## 响应分发器

​	**核心功能**，将原方案中通信层**响应解析(**字符串匹配**)**的职责拆分出来，形成一个单独的，可扩展的响应处理层。
​	设计思想是**表驱动法**，以**响应头**，如：“+IPD，OK ” 和其**处理函数**为表项，构建一个**处理程序表**。当一行响应进入此层时遍历处理表找到匹配的表头，然后调用相应处理函数，将消息的地址继续传递给**命令控制器**。

## 命令控制器

​	**核心功能**，与**响应分发器**共同构成**命令发送**与**消息解析**的底层框架。命令状态由一个**状态机**维护，状态图如下：

![AT状态机](IDE/H743/0-5_example_ESP-AT_Host/AT状态机.png)

​	此状态机并**不表示模块的状态**而是**主机命令的状态**。需要发生的命令都先会进入**命令队列**等待状态机处理。
​	命令发送与接收是两个不同任务，所有在**响应分发器**中注册的命令头都一直会被处理，实际动作**是否执行**还需结合状态位判断。

## 命令注册(添加新的AT功能)

1. 创建一个**全局/静态**的**AT_Cmd_t**命令体，其中包含：
命令内容：					以“\r\n”结尾的字符串。
需要发送的数据(如果有)：		以“\r\n”结尾的字符串，若无数据为NULL 。
超时时间：					uint32_t型。
数据解析回调(若是数据请求命令)： 应用层解析回调，最终响应的数据行会传入此函数，parser_cb
最终响应回调：				被传入最终响应结果和内容，response_cb
下一条命令地址：			    用于构建命令队列

2. 实现所需的数据解析 **parser_cb** (字符串提取) 和 最终响应处理 **response_cb **(如重发)。

3. 在**响应分发器**的**at_handlers**中注册相应的响应头(如+CIPSNTPTIME:) 和 通用的数据命令响应解析
	回调**handle_Rxdata_process**。虽然回调都是相同的但是响应头一定要正确否则可能调用其他应用
	的处理函数。

4. 在需要时调用**AT_controller_cmd_submit**把该命令提交到命令队列中。

以上是一个**主动命令**的添加方法，若要添加一个**非主动请求**的响应处理，比如**+MQTTSUBRECV:** 只需在
**at_handlers**中注册并实现相应处理回调即可。

## 数据点与设备

上报JSON格式：

``` json
{
	"id":"123","version":"1.0","params":{
	"currentTemperature":{"value":22,"time":1747458287111},
	"currenthumidity":{"value":33,"time":1747458287111},
	"标识符":{"value":<指定的数据类型>,"time":<毫秒时间戳>}
	...(一次可上报多个数据点 中间用','隔开)
	}
}
```

​	对比原方案基本无改动，为物理设备提供通用**数据接口**，连接物理设备和MQTT功能，对下提供**数据点构建/初始化**，**数据点修改**；**模块构建/初始化**，**数据点添加**，**模块数据点修改**，**内存释放**功能。对上提供**模块数据JSON序列化**功能。

## 具体功能实现



### WiFi连接功能

​	维护两个私有变量，模块**工作模式**和**wifi状态**。对外提供**初始化**(可设置连接状态改变回调)，**连接AP**，**断开AP**，
**设置工作模式**，**获取工作模式**，**获取wifi状态**。初始化时可注册**状态改变回调**，当wifi连接或断开时，可通知其他模块比如MQTT自动连接或断开清理CFG。

---

### SNTP校准RTC

​	维护两个私有变量，**模块NTP时间就绪**(设置服务器后模块需要一段时间来获取NTP时间)，主机RTC已校准。对外提供**设置SNTP服务器地址**函数(最多设置三个NTP服务器, 最少设置一个有效且可用的服务器)，**查询模块内NTP时间并校准RTC**，**获取RCT校准状态**，**时区设置**，**时间戳获取**功能。注意查询的时间是wifi模块内部的时间，并不是直接从服务器获取的，所以模块如果断电时间就会不准，需要重新获取NTP更新时间，还未实现此功能。

### MQTT相关功能

​	**推送** 功能经**AT_controller_cmd_submit**交由命令控制层完成。
​	**接收订阅** 信息由响应分发器的 **< "+MQTTSUBRECV:"，MQTT_handle_urc_recv >** 处理项接收并处理，响应分发器匹配到响应头后在处理想函数中**解析是哪一个主题的信息**，然后调用相应处理。比如若是 **…property/set** 也就是**设备属性设置主题**的信息，就交由**云命令分发器**处理。
<u>对外提供：</u>
​	**MQTT Broker连接**，**断开**，**获取连接状态**。
​	**MQTT_subscribe**，**MQTT_publish**						   // AT模块通用函数
​	**设备名称(标识符)设置**，**本机名称获取****，**传感器数据推送**		 // 这些根据oneNET功能定义
​	其他功能模块内部调用的函数，MQTT_send_reply



#### 云命令分发器

​	**设备**与MQTT**云设置功能的接口**(依赖**cJSON**库)，内部维护一个**功能点处理表 handler_registry**，表项结构为：
``` c
typedef struct {
    const char* identifier;		//功能点标识符(onenet定义)
    cloud_cmd_handler_t handler;//处理函数
} Cloud_Handler_t;
```

与Json载荷的**"params"**段对应。对设备提供**功能点注册函数**，设备需指定自己的**功能点标识符**(每个设备可有多个可设置功能，需要与onenet定义相同)，和相应**处理函数**。
	在分发器收到**set**主题的一条信息时，会依次拿着**params**段的每个**条目**去遍历**handler_registry**。匹配上对应功能点标识符后调用其处理函数，并将**此条目**传递给设备的处理函数。设备拿到自己的条目就可提取数据使用了。
