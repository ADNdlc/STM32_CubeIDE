# 软件架构设计



## 硬件和平台解耦

实现需求：

1. 更换硬件连接，应用层无需改变，只需修改board_config(板级抽象)

2. 更换芯片，业务代码无需改变，只需做interface对应平台实现和移植

	

一个**成熟、解耦**的软件架构，核心在于**“依赖倒置”**和**“接口隔离”**。高层模块(稳定)不应该依赖于低层模块(变化)，二者都应该依赖于抽象(稳定)。抽象(稳定)不应该依赖于实现细节(变化），实现细节应该依赖于抽象(稳定)。使用封装来创建对象之间的分界层，让设计者可以在分界层的侧进行修改，而不会对另一侧产生不良的影响，从而实现层次间的松耦合。

我们要实现的目标架构如下：

| 层级名称               | 英文名                 | 职责描述                                                     | 典型代码                                | 依赖对象                                    |
| ---------------------- | ---------------------- | ------------------------------------------------------------ | --------------------------------------- | :------------------------------------------ |
| **1. 业务逻辑层**      | **App**                | 纯粹的产品逻辑。只处理数据和状态，**不包含任何硬件定义**。   | statemachine.c, algo_pid.c              | 仅依赖 **Service** 和 **Device** 的抽象接口 |
| **2. 服务组件层**      | **Service**            | 跨模块的系统级功能或复杂的组合逻辑。                         | sys_log.c, mqtt_service.c, data_model.c | 依赖 Device                                 |
| **3. 设备驱动层**      | **Device / Component** | **与板无关**的芯片驱动。只知道协议（I2C/SPI），不知道引脚。  | drv_aht20.c,     drv_esp8266.c          | 依赖 **Platform Interface**                 |
| **4. 工厂/板级配置层** | **Board / BSP**        | 负责实例化驱动，并将其映射到逻辑ID,。是解耦的关键，连接了硬件驱动和逻辑对象。 | board_config.c, board_map.h             | 依赖 Platform 和 Device                     |
| **5. 平台抽象层**      | **Platform / PAL**     | 统一标准接口（I2C/UART/GPIO）。封装 RTOS 和 HAL 差异。       | pal_i2c.c, pal_uart.c                   | 依赖 HAL / LL                               |
| **6. 硬件层**          | **Hardware / HAL**     | 底层硬件操作，直接关联寄存器                                 | HAL库, LL库                             | —                                           |

### 例：创建一个串口队列，并做stm32、ch32的实现

#### 1. 组件(Component)

串口队列的FIFO一般使用**环形缓冲**，环形队列（也称循环队列）是一种特殊的队列，它是一个固定大小的数组，通过特定的算法使得队列的头尾在内存上实现连接。环形队列的特点是当队列满时，新的数据会覆盖最早的数据，从而实现循环利用内存无需手动释放。但当负载过大(写指针超过读指针)时，如一次发送很多日志信息，会造成数据覆盖，导致较早的信息丢失损坏，此时可增大缓冲区大小或做其他适当处理。
此模块是纯算法组件不依赖硬件，我创建在Component文件夹中，基本致结构如下：

```c
// 环形缓冲区
typedef struct {
  uint8_t *buffer;
  size_t size;
  volatile size_t head; // 写指针
  volatile size_t tail; // 读指针
} ring_buffer_t;
ring_buffer_t* rb_create(uint8_t *buffer, size_t size);
```

#### 2. 驱动接口(interface)

**接口**是一种抽象类型，用于定义一组方法规范，而不提供具体实现。它的核心作用是为类的行为制定契约，确保实现接口的类遵循统一的规范。

```c
// 事件类型定义
typedef enum{
  USART_EVENT_RX_COMPLETE,
  USART_EVENT_TX_COMPLETE,
  USART_EVENT_ERROR,
  USART_EVENT_RX_EVENT,
} usart_event_t;

// 回调函数原型
// context:上下文, event:事件类型, args:事件参数(如接收到的数据长度等)
typedef void (*usart_callback_t)(void *context, usart_event_t event, void *args);

// USART 操作接口
typedef struct{
  int (*transmit)(usart_driver_t *self, const uint8_t *data, size_t size,uint32_t timeout);
  int (*receive)(usart_driver_t *self, uint8_t *buffer, size_t size,uint32_t timeout);
  int (*transmit_asyn)(usart_driver_t *self, const uint8_t *data, size_t size);
  int (*receive_asyn)(usart_driver_t *self, uint8_t *buffer, size_t size);
  int (*set_callback)(usart_driver_t *self, usart_callback_t cb, void *cb_context);
} usart_driver_ops_t;

// USART 驱动基类
struct usart_driver_t{
  const usart_driver_ops_t *ops;
};
```
